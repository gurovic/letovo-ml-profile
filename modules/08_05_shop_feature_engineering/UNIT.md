## 2. Зачем существует модуль

**Проблема:** модели из модулей 2 и 4 работают на «как есть» столбцах; без осознанного создания и проверки признаков качество и воспроизводимость падают.

## 5. Результаты обучения

После модуля учащийся может:

1. **Различить** числовые и категориальные признаки и способ их кодирования.
2. **Применить** lambda и `.apply` для построчного преобразования.
3. **Вычислить** групповые агрегаты (`groupby`).
4. **Логировать** шаги preprocessing без декораторов.
5. **Валидировать** вход pipeline через `raise` при нарушении контракта.
6. **Создать** новый признак с обоснованием для задачи магазина.
7. **Собрать** цепочку preprocessing, совместимую с моделью из модулей 2/4.

## 12. Итоговый артефакт

| Поле | Значение |
|---|---|
| Тип | Python-модуль или ноутбук: `preprocess_orders(df) -> df_features` + лог |
| Описание | Цепочка preprocessing для таблицы заказов магазина; валидация; лог шагов |
| ML-смысл | Готовые признаки для регрессии/классификации в следующих модулях |
| Итоговое оценивание | Pipeline воспроизводим; `raise` на невалидных данных; лог содержит порядок шагов |
| Связь с ключевой идеей | Артефакт — инженерный preprocessing, не разовый скрипт |
