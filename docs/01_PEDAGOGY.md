# Pedagogy

Статус: Draft 2

## Назначение

Pedagogy отвечает на один вопрос:

**Как устроено обучение в профиле «Машинное обучение»?**

Документ фиксирует правила принятия педагогических решений. Он не описывает содержание модулей и не дублирует Foundation.

Foundation определяет **каким должен быть выпускник**. Pedagogy определяет **как программа к этому приходит**.

---

## 1. Модель обучения

### Что считается усвоением

Усвоение в профиле — это **наблюдаемое действие**, а не декларация понимания.

Усвоение зафиксировано, если учащийся:

1. выполнил интеллектуальную работу (решил, выбрал, сравнил, обосновал);
2. создал **артефакт**;
3. может **объяснить** своё решение и его ограничения.

### Базовый цикл обучения

Каждый существенный фрагмент программы следует циклу:

```
проблема → гипотеза → действие → артефакт → рефлексия → улучшение
```

Правила:

- без **проблемы** не начинается обучение;
- без **действия** урок не состоялся;
- без **артефакта** модуль не завершён;
- без **рефлексии** опыт не превращается в компетенцию.

### Что не считается обучением

- пассивное слушание без последующей работы;
- копирование кода без решений;
- выполнение инструкций без выбора подхода;
- получение готового ответа от генеративного ИИ без понимания и проверки.

## 2. Формирование компетенций

### Принцип

Компетенции из Foundation (раздел 3) формируются **через повторяющуюся практику**, а не через однократное прохождение темы.

Однократное введение + демо-упражнение **недостаточно** для навыка программирования (Foundation §5, принцип 8).

### Механизмы формирования

| Механизм | Правило |
|---|---|
| Спираль | Каждая ключевая идея возвращается на новом уровне сложности (8→11) |
| Отработка | После введения конструкции/алгоритма — серия задач на тот же навык (на уроке и/или отдельной паре КТП); спираль между годами отработку внутри модуля не заменяет |
| Артефакт | Каждый модуль закрепляет 1–2 компетенции через проверяемый результат |
| Решение | Учащийся принимает решения, без которых артефакт не создаётся |
| Обратная связь | Преподаватель и среда (code review, защита) корректируют практику |
| Интеграция | Компетенции связываются между модулями (данные → модель → система) |

### Программирование как навык

Правила (для модулей Python, алгоритмов, кода в data/ML):

1. **Введение ≠ закрытие темы.** Пара «новая конструкция» без последующей отработки — ошибка планирования.
2. **Отработка — серия задач одной цели**, а не одно упражнение в конце объяснения.
3. **Новую тему не начинать**, пока текущий навык не отработан на уровне самостоятельного решения типовых задач (с обратной связью).
4. **Итоговый артефакт модуля** опирается на уже отработанные навыки; артефакт не заменяет первичную отработку.
5. При нехватке часов — **сократить число тем**, не отработку (Foundation: глубина важнее покрытия).

### Связь с траекторией

| Класс | Фокус формирования компетенций |
|---|---|
| 8 | Python (функции), EDA, первые ML-модели, основы CV |
| 9 | ООП, формализация (вероятность, производные), углубление ML/CV/NLP |
| 10 | ML на базе высшей математики, командная разработка, системные навыки |
| 11 | Самостоятельное проектирование, capstone |

Детали содержания по классам — в Foundation (раздел 7), не в Pedagogy.

## 3. Самостоятельность

Самостоятельность нарастает по классам. Это **минимум**, не потолок.

| Класс | Уровень | Что учащийся делает сам |
|---|---|---|
| 8 | направляемая | Реализует по структуре; выбирает внутри задачи (параметры, визуализации, метрики) |
| 9 | частично самостоятельная | Формулирует подзадачи; проектирует фрагменты решения; обосновывает выбор |
| 10 | командная + самостоятельная в роли | Распределяет работу; принимает архитектурные решения в команде; отвечает за свой модуль |
| 11 | самостоятельная | Сам ставит задачу capstone; планирует, реализует, защищает |

### Правило

Если учащийся не принимает **ни одного самостоятельного решения** на уроке — урок перепроектируется (см. Material Design Standard).

## 4. Проекты в программе

### Типы проектной работы

| Тип | Где | Срок | Результат |
|---|---|---|---|
| Модульный артефакт | каждый существенный модуль | 2–6 недель | законченный фрагмент (ноутбук, модель, библиотека) |
| Командный проект | 10 класс | семестр или модуль | общий репозиторий, распределённая работа |
| Capstone | 11 класс | год или полугодие | самостоятельный проект с защитой |

### Правила проектов

1. **Проект ≠ большое домашнее задание.** У проекта есть постановка задачи, ограничения и критерии успеха.
2. **Командный проект обязателен в 10 классе** (Foundation, раздел 8).
3. **Capstone обязателен в 11 классе** — индивидуальный или с явно разделёнными ролями.
4. Исследовательский и инженерный проекты равноправны; выбор зависит от задачи, не от класса.

### Критерий качества проекта

Проект успешен, если учащийся может ответить:

- какую проблему решал;
- какие решения принял и почему;
- что не получилось и что бы изменил;
- какие ограничения у решения.

## 5. Принципы последовательности модулей

Unit Planner описывает **один модуль**. Pedagogy описывает **логику цепочки модулей** на протяжении года и четырёх лет.

### Правила построения последовательности

1. **Пререквизит перед темой.** Модуль не начинается, пока не закрыты пререквизиты (данные, код, математика из спецкурса).
2. **От простого к сложному внутри года.** Сначала инструмент, затем метод, затем система.
3. **Сквозная нить.** Модуль опирается на артефакты предыдущих модулей, а не начинается с нуля.
4. **Один модуль — одна главная идея.** Если идей несколько — модуль делится.
5. **Проектная точка.** Каждый год содержит хотя бы один модуль, где артефакт интегрирует несколько навыков.
6. **Буфер на повторение.** В каждом году закладывается время на закрепление и доработку, а не только на новые темы.
7. **Отработка в плане года.** В КТП для программных тем видны пары/блоки практики, а не только названия новых конструкций.

### Решение при проектировании годовой программы

Перед утверждением набора модулей на год проверить:

- [ ] каждый модуль закрывает пререквизиты следующего;
- [ ] есть нарастание самостоятельности;
- [ ] есть хотя бы один интеграционный артефакт;
- [ ] нагрузка реалистична (глубина важнее количества);
- [ ] у программных тем заложена отработка (не только введение);
- [ ] траектория класса согласована с Foundation (раздел 7).

## 6. Артефакты в обучении

Foundation описывает **систему артефактов и портфолио**. Pedagogy описывает **как артефакты работают в процессе обучения**.

### Роли артефакта

| Роль | Правило |
|---|---|
| Фиксация | Артефакт фиксирует результат, а не намерение |
| Обратная связь | По артефакту даётся конкретная обратная связь |
| Прогресс | Артефакты разных лет показывают рост сложности |
| Оценивание | Артефакт — основной объект оценивания (раздел 9) |
| Портфолио | Артефакты накапливаются в портфолио (Foundation, раздел 8) |

### Минимум портфолио

См. Foundation, раздел 8. Pedagogy определяет **как** учащийся к этому минимуму приходит — через модульные артефакты, командный проект (10 класс) и capstone (11 класс).

---

## 7. Роль преподавателя

### Преподаватель — не лектор

Преподаватель проектирует **условия для работы учащегося**, а не транслирует знания.

### Когда преподаватель вмешивается

| Ситуация | Действие |
|---|---|
| Учащийся застрял > 10 мин без прогресса | Наводящий вопрос, не готовое решение |
| Ошибка концептуальная | Возврат к пререквизиту или контрпример |
| Ошибка техническая | Подсказка по отладке, не исправление кода |
| Учащийся завершил задачу рано | Усложнение или исследовательский вопрос |
| Групповой проект | Фасилитация, code review, разрешение конфликтов |

### Когда преподаватель не вмешивается

- учащийся пробует альтернативный подход, который может сработать;
- ошибка — часть запланированного обучения (см. expected mistakes в Lesson Design);
- учащийся обсуждает решение с одноклассниками.

### Правило

Преподаватель не пишет код за учащегося и не генерирует решение через ИИ от его имени.

## 8. Использование ИИ учащимися

ML-профиль обучает работе **с** ИИ и **без** него. Это педагогическое решение, не техническое.

### Обязательно без генеративного ИИ

Учащийся должен уметь **самостоятельно**, без генерации:

- написать базовый Python-код (функции, классы, циклы, работа с данными);
- провести EDA и интерпретировать результаты;
- реализовать классический ML-метод;
- объяснить выбор метрики, loss, архитектуры;
- отладить свой код;
- защитить своё решение на защите / capstone.

### Допустимо с генеративным ИИ

| Контекст | Разрешено | Условие |
|---|---|---|
| Поиск документации | да | учащийся проверяет результат |
| Генерация шаблонного кода | да | учащийся понимает и может переписать |
| Рефакторинг | да | учащийся проверяет поведение |
| Генерация тестов | да | учащийся понимает, что тестируется |
| Объяснение ошибки | да | учащийся воспроизводит и исправляет |
| Мозговой штурм | да | финальное решение — своё |

### Запрещено (в рамках профиля)

- сдавать работу, где ключевые решения принял ИИ, а учащийся не может их объяснить;
- использовать ИИ на контрольных и защитах capstone без явного разрешения преподавателя;
- генерировать отчёт или презентацию без понимания содержания.

### Олимпиады и соревнования

Правила использования ИИ **на олимпиадах устанавливают организаторы**, не профиль.

| Контекст | Правило |
|---|---|
| Этапы профиля (уроки, модули, capstone) | Действуют правила этого раздела |
| Олимпиады, соревнования, отборочные | Действует регламент конкретного мероприятия |
| Полный запрет ИИ | Например, этапы Всероссийской олимпиады — следовать регламенту |
| Ограниченный чат | Если регламент разрешает — только в заданных границах |

Pedagogy **не дублирует** олимпиадные регламенты. При подготовке к олимпиаде преподаватель указывает учащимся актуальные правила конкретного мероприятия.

### Правило документирования

Если ИИ использовался — учащийся **явно указывает** это в артефакте (что сгенерировано, что проверено, что переписано). Скрытое использование — нарушение.

## 9. Оценивание

Шкала, веса и связь с аттестацией **фиксированы школьной политикой Летово**. Pedagogy их не определяет.

### Принцип

Оценивается **компетенция**, проявленная в **артефакте**, а не объём выполненной работы.

### Объекты оценивания

| Объект | Когда | Что оценивается |
|---|---|---|
| Артефакт модуля | конец модуля | соответствие цели модуля, качество решения |
| Командный проект | 10 класс | вклад, collaboration, итоговый результат |
| Capstone | 11 класс | полный цикл: постановка → реализация → защита |
| Портфолио | 11 класс | полнота, прогресс, связность |

### Критерии (общие)

Оценивание по каждому артефакту проверяет:

1. **Решение** — учащийся принял и обосновал решения.
2. **Реализация** — код/модель/анализ работает и воспроизводим.
3. **Понимание** — учащийся объясняет, что сделал и почему.
4. **Ограничения** — учащийся знает, где решение не работает.
5. **Коммуникация** — результат оформлен так, что другой может проверить.

### Чего нет в оценивании

- оценки за посещаемость как самостоятельного критерия;
- оценки за объём кода;
- оценки за «красоту» без содержания;
- единого экзамена по ML как замены портфолио.

## 10. Дифференциация

### Принцип

Программа **единая** для всех учащихся профиля. Дифференциация — внутри модулей, без разветвления траектории.

### Механизмы

| Механизм | Слабее базы | Сильнее базы |
|---|---|---|
| Глубина задачи | упрощённый вариант с тем же артефактом | исследовательское расширение |
| Поддержка | больше структуры, меньше открытых решений | меньше подсказок, больше открытых вопросов |
| Aspirational-трек | не обязателен | дополнительные задачи (MLOps, статья, соревнование) |
| Командный проект | чёткая роль с понятным scope | роль архитектора или исследователя |

### Правило

Дифференциция **не снижает** обязательные компетенции Foundation. Она меняет глубину и поддержку, не набор компетенций.

### Калибровка времени

**80 минут** в `LESSON.md` — фиксированная длительность пары (учёт и проведение). Ориентиры по блокам в зоне B рассчитаны на **смешанный** класс: есть ветка «слабее базы», буфер на типичные ошибки, общий темп.

| Ситуация | Правило |
|---|---|
| Однородно сильное ядро (одарённые, сильная база Python) | Активная работа по минимуму сдачи часто **в 2–3 раза короче** заложенного; оставшееся время — углубление (зона C «сильнее базы»), исследовательский вопрос, переход к следующей паре **не** сокращением календаря КТП |
| Смешанный класс | Ориентиры B — норма; этапы «углубление» опциональны |
| Оценка урока | Разделять **нагрузку идеи** (новая мысль) и **нагрузку исполнения** (объём кода, шаблонность задач); сильным часто не хватает второго |

Детальная процедура калибровки и эталон — Lesson Design, раздел «Калибровка пары». Конкретные усиления для пилота — в зоне C соответствующих `LESSON.md`.

## 11. Выбор формата обучения

### Когда использовать формат

| Формат | Использовать, если | Не использовать, если |
|---|---|---|
| Урок (введение) | нужна новая идея; 1–2 управляемых задачи на контракт | тема уже введена и нужна только отработка |
| Урок (отработка) | навык введён; нужна серия самостоятельных задач на ту же тему | одновременно вводится новая несвязанная конструкция |
| Активность | задача решается за 1–3 занятия с конкретным артефактом | нужна длинная исследовательская работа |
| Проект (модульный) | артефакт интегрирует несколько **уже отработанных** навыков | первичная отработка конструкции ещё не сделана |
| Командный проект | нужны навыки совместной разработки | задача слишком мала для команды |
| Самостоятельная работа | закрепление или подготовка к следующему модулю | учащийся ещё не освоил пререквизит |
| Презентация | нужно общее внимание к визуалу / сравнению | учащиеся могут работать с материалом напрямую |
| Ноутбук | нужны эксперименты и итерации | достаточно короткого скрипта |

### Правило

Формат выбирается **после** определения цели (Lesson Design), а не потому что «так принято».
