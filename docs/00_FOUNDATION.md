# Foundation

Статус: Draft 2 (утверждён с оговорками: раздел 6 — на согласовании; раздел 7 — TODO по 11 классу)

## Назначение

Foundation — самый стабильный документ проекта. Он отвечает на один вопрос:

**Зачем существует профиль и каким он должен быть?**

Foundation фиксирует решения, которые меняются крайне редко. Он не описывает модули, уроки, методы обучения и шаблоны документов.

При противоречии между документами приоритет у Foundation.

---

## 1. Проблема

Школьники 8–11 классов, интересующиеся машинным обучением, обычно получают либо поверхностное знакомство с инструментами, либо фрагментарные знания без инженерной практики.

Профиль «Машинное обучение» решает другую задачу: за четыре года сформировать учащегося, который умеет **самостоятельно проектировать, реализовывать и обосновывать решения на стыке программирования, анализа данных и машинного обучения** — на уровне, достаточном для осознанного продолжения в вузе и профессиональной среде.

---

## 2. Область действия этого документа

Foundation описывает **профиль «Машинное обучение»** — образовательную программу 8–11 классов.

### Входит в область профиля

- программирование (продолжение Python; смежные технологии);
- анализ данных;
- машинное обучение;
- смежные инженерные темы (веб-разработка, базы данных и аналогичные);
- инженерная практика: артефакты, проекты, командная работа.

### Не входит в область этого документа

| Компонент | Правило |
|---|---|
| Курс спецматематики | Отдельный курс. Профиль **использует** его результаты, но не проектирует его. |
| Курс экономики | Отдельный курс. Не рассматривается. |
| Основы Python в 7 классе | Пререквизит. Профиль **продолжает** Python, но не проектирует начальное обучение. |

## 3. Портрет выпускника

Выпускник профиля (11 класс) — учащийся, который:

1. **Программирует** как инженер: пишет читаемый код, работает с функциями, классами, инструментами командной разработки, понимает ограничения технологий.
2. **Работает с данными**: проводит EDA, готовит данные, визуализирует, формулирует и проверяет гипотезы.
3. **Применяет ML**: понимает и реализует классические методы, обосновывает выбор подхода, интерпретирует результаты, знает границы применимости.
4. **Мыслит математически** в контексте задачи: использует линейную алгебру, анализ, вероятность и статистику там, где они нужны для решения — не как абстрактный курс, а как инструмент инженера.
5. **Создаёт артефакты**: код, ноутбуки, модели, отчёты, презентации — и может показать связную траекторию своей работы за четыре года.

### Компетенции

Компетенции сформулированы как **наблюдаемые действия**. Категории:

- **обязательные** — без них выпускник не считается сформированным;
- **профильные** — ожидаются у большинства, допускают вариативность глубины;
- **aspirational** — для учащихся, выходящих за базовый уровень.

## 4. Границы профиля

### Профиль — это

- инженерная программа на стыке программирования, данных и ML;
- программа, где учащийся **делает**, а не только слушает;
- программа с нарастающей математической строгостью;
- программа, готовящая к осознанному выбору дальнейшего пути (вуз, исследование, инженерия).

### Профиль — это не

- курс спецматематики (математика — параллельный курс, не заменяется);
- курс экономики;
- олимпиадная подготовка по ML (хотя олимпиадные задачи могут быть контекстом);
- набор ноутбуков без инженерной логики;
- курс лекций по алгоритмам без реализации и экспериментов;
- обзор нейросетей без понимания классического ML и данных.

## 5. Принципы программы

Каждый модуль и каждый урок профиля должны соответствовать этим принципам. Если модуль им противоречит — модуль перепроектируется.

1. **Инженерная практика важнее покрытия тем.** Лучше глубже пройти меньше, чем поверхностно «охватить всё».
2. **Артефакт важнее усвоенной лекции.** Учащийся должен что-то создать, а не только что-то узнать.
3. **Решение важнее инструкции.** Учащийся принимает решения: выбирает подход, обосновывает, сравнивает альтернативы.
4. **Математика — инструмент задачи, а не самоцель.** Математические идеи вводятся тогда, когда без них нельзя двигаться дальше в профиле.
5. **Прогрессия обязательна.** Каждый год опирается на предыдущий; нельзя переносить сложность вперёд без пререквизитов.
6. **Профессиональный контекст обязателен.** Учащийся понимает, зачем изучает идею, а не только как её реализовать.
7. **Простота архитектуры важнее полноты.** Не добавлять темы, форматы и сущности без явной необходимости.
8. **Навык программирования — через отработку.** Введение конструкции или алгоритма и одно упражнение не закрывают тему: нужна серия практических задач на тот же навык до устойчивого самостоятельного исполнения.

## 6. Принципы построения содержания

> **Статус:** черновик; детальное согласование отложено.

Содержание профиля строится не как список тем, а по следующим правилам:

| Принцип | Правило |
|---|---|
| От данных к модели | Сначала данные и их свойства, затем метод. |
| От идеи к математике | Сначала инженерная задача, затем формализация. |
| От реализации к пониманию | Сначала работающий прототип, затем анализ поведения и ограничений. |
| От классического к современному | Классический ML и EDA — фундамент; современные методы опираются на него. |
| От индивидуального к командному | Сначала личная инженерная работа, затем командная разработка. |
| От введения к отработке | Сначала контракт/идея и 1–2 управляемых задачи, затем серия самостоятельных задач на тот же навык; одно упражнение ≠ освоение. |
| От упражнения к исследованию | Сначала задачи с известным ответом, затем открытые вопросы. |
| От инструмента к системе | Сначала навык, затем его место в архитектуре решения. |
| От реальных ограничений | Задачи опираются на реальные или реалистичные данные и ограничения, а не только на синтетические примеры. |
| От итерации | Каждое решение можно улучшить; возврат и доработка — нормальная часть работы. |
| Сквозная связность | Модуль опирается на артефакты предыдущих модулей, а не начинается с нуля. |
| Этика по необходимости | Вопросы смещения данных, приватности и ответственности ML вводятся там, где без них нельзя принять решение. |
| Олимпиады и соревнования — контекст | Могут быть мотивацией и площадкой, но не определяют архитектуру программы. |

## 7. Траектория 8–11

Траектория описывает **развитие выпускника**, а не календарь тем. Ниже — зона ответственности **курса профиля** (без спецматематики и экономики).

### Общая логика четырёх лет

```
8 класс   → инженер-исследователь данных: Python, EDA, первые ML-модели
9 класс   → инженер с формализацией: ООП, вероятность и производные в задачах, углубление ML и CV
10 класс  → инженер в команде: ML на базе высшей математики, командная разработка, системные навыки
11 класс  → инженер-практик: продвинутый ML, системное мышление, capstone-проект
```

### Нагрузка профиля

Академические часы профиля «Машинное обучение» в неделю (7 класс — пререквизит, 2 ч/нед, вне этой таблицы):

| Класс | Часов в неделю |
|---|---|
| 8 | 5 |
| 9 | 5 |
| 10 | 6 |
| 11 | 6 |

**Правило планирования:** 1 урок = 1 пара = 2 академических часа (учёт нагрузки) = **80 минут** проведения. Длительность пары **всегда** 80 минут; ориентиры времени в `LESSON.md` суммируются в 80. При 34 учебных неделях: 8 класс — 85 пар (модуль 1 включает вводную пару); 9 класс ≈ 85 пар/год; 10–11 класс ≈ 102 пары/год.

### 8 класс

**Кто становится учащийся:** инженер, который умеет писать структурированный код и самостоятельно исследовать данные.

**Программирование:** функции (продолжение Python после 7 класса); **базовые алгоритмы** — обязательный блок (см. таблицу).

**Базовые алгоритмы 8 класса** — три последовательных блока. Нужны для ЕГЭ по информатике, технических собеседований и олимпиад по ML/AI (перебор, DP, работа с данными в памяти).

| Блок | Содержание | Граница |
|---|---|---|
| Массивы | линейный и бинарный поиск; сортировки (в т.ч. рекурсивные); интуиция сложности O(n²) vs O(n log n) | 8 класс |
| Структуры данных | стек, очередь, deque, множество (set); словарь как hash map | **куча (heap) — 9 класс** |
| Динамическое программирование | мемоизация, табличный DP; **финал блока — игровые задачи** (ходы, win/lose) | 8 класс |

Блоки проектируются как **отдельные модули** (Unit Planner), не как один урок. Итоговая работа модуля — **до 2 уроков** сверх уроков содержания.

**Данные и ML:** полное освоение EDA; техническая часть классического ML; основы computer vision.

**Смежные темы:** по мере необходимости для задач модуля (веб, базы данных и аналогичные).

**Переход к 9 классу:** от «я могу реализовать и исследовать» к «я могу формализовать и обосновать».

### 9 класс

**Кто становится учащийся:** инженер, который строит абстракции (классы) и использует математику в контексте задачи.

**Программирование:** классы и объектно-ориентированное программирование; **куча (heap, heapq)** — приоритетная очередь, топ-K (пререквизит: стек, очередь из 8 класса).

**Данные и ML:** углубление EDA, классического ML, CV и NLP; использование вероятностей и производных там, где это необходимо.

**Переход к 10 классу:** от индивидуальной работы к командной; от ML «без строгости» к ML на базе высшей математики.

### 10 класс

**Кто становится учащийся:** инженер в команде, который применяет ML с опорой на высшую математику.

**Программирование:** технологии командной разработки.

**Данные и ML:** ML с полной опорой на базу спецматематики.

**Смежные темы:** веб, базы данных и аналогичные — в контексте системных задач.

**Переход к 11 классу:** от командного исполнения к самостоятельному проектированию и исследованию.

### 11 класс

## 8. Система артефактов

За четыре года учащийся производит набор артефактов, который образует **портфолио профиля**.

### Типы артефактов

| Тип | Примеры |
|---|---|
| Код | скрипты, библиотеки, модули |
| Данные | подготовленные датасеты, документация к данным |
| Аналитика | ноутбуки EDA, отчёты |
| Модели | обученные модели, эксперименты, логи |
| Коммуникация | презентации, статьи, публичные выступления |
| Проекты | законченные инженерные и исследовательские работы |

### Правила системы артефактов

1. Каждый существенный модуль завершается артефактом.
2. Артефакты накапливаются: учащийся может показать прогресс за четыре года.
3. Артефакт **проверяемый** — не «я понял», а «вот результат, вот обоснование».
4. Артефакты разных лет демонстрируют нарастание сложности.

### Обязательный минимум портфолио к выпуску

| # | Требование | Зачем |
|---|---|---|
| 1 | Не менее **одного артефакта на каждый год** (8, 9, 10, 11), демонстрирующего рост сложности | Траектория видна |
| 2 | **EDA-проект**: ноутбук или отчёт с полным циклом исследования данных и выводами | Базовая data-компетенция |
| 3 | **ML-проект**: от данных до модели с обоснованием выбора метода и метрик | Базовая ML-компетенция |
| 4 | **Командный проект** (10 класс): общий репозиторий, распределённая работа, итоговый артефакт | Инженерная практика |
| 5 | **Capstone** (11 класс): самостоятельный проект с постановкой задачи, реализацией, отчётом и защитой | Портрет выпускника |
| 6 | **Лог экспериментов** хотя бы по одному ML-проекту: что пробовал, что изменил, что получил | Культура эксперимента |

Механизм использования артефактов в обучении и связь с оцениванием — в Pedagogy.

## 9. Критерии успеха программы

Программа считается успешной, если через четыре года выполнены условия:

1. **Выпускник соответствует портрету** (раздел 3) — через портфолио и capstone, а не через единый экзамен.
2. **Траектория реализована** — пройдены качественные переходы 8→9→10→11.
3. **Инженерная культура сформирована** — код, данные, эксперименты, команда.
4. **Математика интегрирована** — используется в задачах профиля, а не изолированно.

Количественные метрики успеха программы **не зафиксированы**; будут определены позже.

## 10. Антипаттерны

| Антипаттерн | Признак | Действие |
|---|---|---|
| Курс лекций | Урок = объяснение без работы учащегося | Перепроектировать урок |
| Набор ноутбуков | Материалы не связаны инженерной логикой | Пересмотреть модуль |
| Гонка за темами | Новая тема каждую неделю без глубины | Сократить содержание |
| Разовое упражнение = навык | Тема «пройдена» после одного демо-задания | Добавить отработку (серия задач / пара практики); убрать следующую тему |
| ML без данных | Алгоритмы без EDA | Вернуть этап работы с данными |
| ML без математики (10–11) | Эвристики вместо понимания | Вернуть опору на спецматематику |
| Математика без задачи | Формулы без инженерного контекста | Привязать к задаче профиля |
| Имитация проектов | Проект = большое домашнее задание | Пересмотреть архитектуру проекта |
| Замена инженерии ИИ | Учащийся не может решить задачу без генеративного ИИ | Пересмотреть требования (Pedagogy) |
| Смена языка ради новизны | 11 класс = другой язык без ML-цели | Вернуть Python; новизну давать через содержание |
