# Material Design Standard

Статус: Draft 2

## Назначение

Material Design Standard отвечает на один вопрос:

**Как должны проектироваться образовательные материалы?**

Документ задаёт **обязательные правила** для любого материала профиля: модуля, урока, ноутбука, презентации, активности, проекта.

Шаблоны конкретных сущностей — в Unit Planner, Lesson Design и форматных стандартах. Здесь — только правила, общие для всех.

---

## 1. Принцип проектирования

Материал проектируется вокруг **деятельности учащегося**, а не объяснения преподавателя.

Единица проектирования — не содержание, а **интеллектуальная работа**, которую выполняет учащийся.

Материал оправдан, если после него учащийся **может делать то, чего не мог раньше**.

---

## 2. Обязательные вопросы перед созданием материала

Перед созданием **любого** материала автор отвечает на десять вопросов:

| # | Вопрос |
|---|---|
| 1 | Зачем этот материал нужен? |
| 2 | Почему он стоит именно здесь в программе? |
| 3 | Какие пререквизиты предполагаются? |
| 4 | Какую новую способность получит учащийся? |
| 5 | Какая инженерная практика развивается? |
| 6 | Какие математические идеи нужны? |
| 7 | Какие решения примет учащийся? |
| 8 | Какой артефакт создаст учащийся? |
| 9 | Какие типовые ошибки ожидаются? |
| 10 | Как будет признан успех? |

Если на вопрос нет убедительного ответа — материал перепроектируется.

---

## 3. Результаты обучения

Результаты обучения описывают **наблюдаемые действия**.

Допустимые глаголы:

- реализует…
- объясняет…
- сравнивает…
- проектирует…
- анализирует…
- оценивает…
- улучшает…

Недопустимые формулировки:

- знает…
- понимает…
- знакомится с…

---

## 4. Центральная идея

У каждого урока или активности — **ровно одна** центральная идея.

Вспомогательные идеи допустимы, но только если усиливают центральную.

Если несколько несвязанных идей конкурируют за внимание — материал **делится**.

---

## 5. Работа учащегося

Учащийся регулярно:

- реализует код;
- исследует поведение систем;
- отлаживает;
- сравнивает альтернативы;
- обосновывает решения;
- коммуницирует результаты;
- улучшает предыдущую работу.

Материалы, где учащийся в основном потребляет контент, **недопустимы**.

### Отработка навыка программирования

Для материалов, где результат — умение писать код (конструкция, алгоритм, API данных):

| Правило | Смысл |
|---|---|
| Введение отдельно от объёма отработки | 1–2 управляемых задачи показывают контракт; навык даёт **серия** задач той же цели |
| Одно упражнение недостаточно | Закрытие темы после одного упражнения-показа — нарушение Foundation §5 (принцип 8) |
| Задачи одной цели | На отработке не смешивать новую несвязанную конструкцию |
| Проверяемость | У каждой задачи — критерий успеха (тест, ожидаемый вывод, чек-лист) |

Отработка может быть блоком внутри урока или отдельной парой КТП («Практика: …»). Выбор — по объёму навыка (Pedagogy §11).

---

## 6. Интеллектуальные решения

Каждый материал требует **осмысленных решений** от учащегося.

Примеры:

- выбор признаков;
- выбор метрик;
- проектирование эксперимента;
- выбор алгоритма;
- интерпретация результатов;
- улучшение решения.

Следование инструкции без решений — **недостаточно**.

### Задачи повышенной сложности

Помимо закрепления и типовых задач, в материалах **регулярно** дают задачи с нетривиальным **логическим, математическим или алгоритмическим** содержанием: контрпример, обоснование «да/нет», поиск условия, сравнение критериев, устойчивость вывода.

| Правило | Пояснение |
|---|---|
| Не вместо базы | Сильные задачи **дополняют** закрепление, не заменяют его |
| На паре и в ДЗ | Хотя бы одна такая задача в **ДЗ**; на паре — по объёму (вопрос, эксперимент без готового ответа) |
| Интерес → в ДЗ | Самые увлекательные задачи (вызов, неожиданный поворот) — **в домашнее**, чтобы была мотивация решать дома; на паре — ядро навыка ([Lesson Design](04_LESSON_DESIGN.md) § «Домашнее задание») |
| Проверяемый ответ | Контрпример, число, короткое обоснование — не «рассуждение вообще» |
| Уровень класса | Без олимпиадного формализма; опора на пройденные инструменты |

---

## 7. Инженерные артефакты

Каждый существенный урок производит **не менее одного артефакта**.

Типичные артефакты:

- исходный код;
- ноутбук;
- отчёт;
- презентация;
- датасет;
- библиотека;
- лог экспериментов;
- документация проекта.

Артефакты должны быть близки к профессиональной практике, насколько это возможно в школьных условиях.

---

## 8. Исследовательский компонент

Где уместно — включать **хотя бы один открытый вопрос** без заранее заданного ответа.

Примеры:

- Какой метод работает лучше и почему?
- Что происходит при нарушении допущений?
- Можно ли улучшить baseline?
- Какая гипотеза подтверждается данными?

---

## 9. Типовые ошибки

Автор **явно фиксирует**:

- ожидаемые заблуждения;
- ошибки реализации;
- математические недопонимания;
- неверные инженерные решения.

Материал помогает учиться **на ошибках**, а не полностью их избегать.

---

## 10. Профессиональный контекст

Автор указывает, **где изучаемая идея встречается** в профессиональной ML-практике.

Учащийся должен понимать, **зачем** он это изучает.

---

## 11. Чек-лист качества материала

Перед публикацией проверить:

- [ ] ясная цель;
- [ ] одна центральная идея;
- [ ] достаточные пререквизиты;
- [ ] наблюдаемые результаты обучения;
- [ ] осмысленные решения учащегося;
- [ ] инженерный артефакт;
- [ ] для кода: отработка навыка (серия задач), не одно упражнение;
- [ ] исследовательский элемент (где уместно);
- [ ] профессиональная релевантность;
- [ ] согласованная терминология;
- [ ] связность с предыдущими материалами.

Провал по любому пункту — доработка.

---

## 12. Границы документа

| В Material Design Standard | Не в Material Design Standard |
|---|---|
| Правила проектирования любого материала | Шаблон модуля → Unit Planner |
| Формат результатов обучения | Шаблон урока → Lesson Design |
| Общий чек-лист качества | Ноутбук → Notebook Standard |
| | Презентация → Presentation Standard |

При противоречии с Foundation или Pedagogy — приоритет у них.
