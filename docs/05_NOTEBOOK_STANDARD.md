# Notebook Standard

Статус: Draft 2

## Назначение

Notebook Standard отвечает на один вопрос:

**Как проектируется учебный Jupyter-ноутбук?**

Ноутбук — интерактивная среда обучения, а не лекция и не репозиторий кода.

Общие правила материалов — в Material Design Standard. Контекст урока — в Lesson Design. Здесь — **правила формата ноутбука**.

---

## Когда нужен ноутбук

Ноутбук создаётся, если урок требует:

- пошаговых экспериментов;
- итеративной работы с кодом и данными;
- фиксации наблюдений по ходу работы.

Если достаточно скрипта или активности без пошагового сопровождения — ноутбук не обязателен (см. Pedagogy, раздел 11).

---

## 1. Цели формата

Ноутбук помогает учащемуся:

- думать;
- экспериментировать;
- реализовывать;
- исследовать;
- фиксировать выводы.

(Общий принцип «деятельность, а не потребление» — Material Design Standard, раздел 5.)

---

## 2. Структура ноутбука

Ноутбук — **рабочая среда**, не конспект. Начинается сразу с работы.

| # | Раздел | Содержание |
|---|---|---|
| 1 | Заголовок | Одна строка `# Название урока`. Без «Пара КТП», «Идея», «Данные», «Результаты обучения», «Пререквизиты» |
| 2 | Данные / импорты | **Сразу после заголовка.** Минимальные импорты или данные урока inline (см. §3) |
| 3 | Рабочие блоки | Короткие исполняемые ячейки |
| 4 | Направляемые эксперименты | С изменением параметров |
| 5 | Вопросы для рефлексии | Короткие, по делу |
| 6 | Самостоятельное задание | Расширение, не повторение |
| 7 | Дополнительно | По желанию, для сильных |

**Не помещать в ноутбук:**

| Убрать из ноутбука | Куда переносится |
|---|---|
| Цель, идея, роль пары, «Пара КТП N» | `LESSON.md` зона A/D |
| Результаты обучения | `LESSON.md` §D, MDS §3 |
| Пререквизиты | `LESSON.md` зона A |
| «Итог», «Дальше — пара N» | `LESSON.md` зона A (поле «Дальше») |

Между заголовком и первой ячейкой импортов **не должно быть** markdown-вставок.

## 3. Код

| Правило | Пояснение |
|---|---|
| Короткие ячейки | Одна ячейка — одна мысль |
| Одна цель на ячейку | Не смешивать загрузку, анализ и визуализацию |
| Без магии | Нет необъяснённого кода; каждая функция мотивирована |
| Скрытая сложность | Вспомогательное — в функции или импорте, не в теле урока |
| Воспроизводимость | Ноутбук выполняется сверху вниз без ручных правок |

### Деятельность учащегося (не прогон готовых ответов)

Учебный `lesson.ipynb` / `homework.ipynb` — место, где учащийся **пишет и проверяет** код. Эталон глубины — пары модуля `08_01` (напр. `02_function_as_mapping`): stubs + конкретные `assert` + вызов в ДЗ.

| Правило | Пояснение |
|---|---|
| Stub, не ответ | Рабочие ячейки урока/ДЗ — `pass` / пустые переменные / `# ваш код`; полный ответ — только в `solutions.ipynb` |
| Assert задаёт контракт | После stub — форма/диапазон/тип («не пусто», `len`, dtype), не готовая формула решения |
| Assert ДЗ ≠ ответ | В `homework.ipynb` **запрещено** писать в `assert` вычисление того же ответа (`== df[…].mean()`, `== (mask).sum()`, формула MAE и т.п.). Такие проверки — только в `solutions.ipynb` |
| Готовый код только как опора | Допустим в стартовой ячейке (данные, уже изученный pipeline); **не** вместо задачи раздела |
| ДЗ = закрепление + вызов | Уровни A–D (Lesson Design); последний блок — не повтор ячейки урока |
| Запрет «scaffold» | Ноутбук, где ученик только Restart & Run All по готовым ответам — **не готов** к публикации / gist / Canvas |

**Нарушение (ДЗ):** `assert n_comfort == (df['vehicle_type'] == 'comfort').sum()` — ответ скопирован в assert.  
**Замена в ДЗ:** `assert n_comfort is not None and 0 < n_comfort < len(df)`; эталон числа — в `solutions.ipynb`.

**Нарушение (урок):** в `lesson.ipynb` уже заполнены `TARGET`, фильтры, `fit`/`predict`, MSE; в `solutions.ipynb` — две ячейки «см. урок».

**Замена:** ученик заполняет stubs; `solutions.ipynb` закрывает **каждый** блок урока и ДЗ (заголовок «Для преподавателя», секции Урок / ДЗ).

### Данные в ноутбуке

Пока объём данных **небольшой** (ориентир: до ~50–100 строк literal в ячейке, один микро-датасет), ноутбук **самодостаточен**: данные — в ячейке сразу после заголовка.

| Правило | Пояснение |
|---|---|
| Inline | Списки, словари, константы урока — literal в коде ноутбука |
| Повтор — не повод выносить | Один и тот же `APARTMENTS` в нескольких ноутбуках модуля — **норма**; копируйте literal, не импортируйте ради DRY |
| Без `sys.path` | Учащийся не настраивает пути к модулю |
| Без скрытого источника | Нет импорта из `data/module_datasets.py`, если набор можно прочитать целиком в одной ячейке |
| Когда вне ноутбука | Только **большой** объём: CSV/файл, сотни+ строк, бинарные данные; загрузка явная (`read_csv`, `open`) |

**Нарушение:** `from data.module_datasets import APARTMENTS` на паре с пятью квартирами — «чтобы не дублировать».

**Замена:** та же пятёрка квартир literal в ноутбуке, даже если она уже есть в другом уроке модуля.

`data/module_datasets.py` (и аналоги) — **канон для автора**, артефакта и сверки; учебный ноутбук **не обязан** на него ссылаться, пока данные малы. При правке сюжета — обновить канон и все inline-копии в ноутбуках этого модуля.

### Пререквизиты в коде

Типы и конструкции в ноутбуке — только из пререквизитов урока и модуля (UNIT §14, Foundation §7). **Канон по парам 8 класса:** [07_KTP.md — Карта инструментов](07_KTP.md#карта-инструментов) → [ktp/08.md](ktp/08.md#карта-инструментов).

| Правило | Пояснение |
|---|---|
| Без забегания | Нет конструкций, которые учащийся ещё не проходил, без отдельного введения |
| dict | До модуля про структуры данных — **не использовать** в учебных ноутбуках; вместо `{'ключ': значение}` — параллельные списки, кортежи, отдельные переменные |
| Явное введение | Если dict нужен раньше программы — отдельная пара с введением, не «по ходу» в чужом синтаксисе |

**Нарушение:** `{'area_sqm': 28, 'price_mln': 3.9}` на паре про функции в 8 классе.

**Замена:** `AREAS_SQM = [28, …]` и `PRICES_MLN = [3.9, …]` с пояснением «один индекс — одна квартира».

## 4. Текст

- Текст объясняет **решения**, а не синтаксис.
- Предпочитать **вопросы** длинным объяснениям.
- Markdown — для постановки задачи и рефлексии, не для лекции.

### Словарь для учащегося

Ноутбук читает **ученик**, а не автор модуля. Текст в ячейках — на языке пререквизитов и текущего модуля, не на языке будущих пар.

| Правило | Пояснение |
|---|---|
| Только известные слова | Термин — из пререквизитов, глоссария модуля (UNIT §18) или вводится в этом уроке |
| Нет мостов для учителя | Сравнения с библиотеками и API из будущих модулей (`sklearn`, `fit`, `model.predict`) — в `LESSON.md` зона D, **не** в ноутбуке |
| Проверка фразы | Если учащийся не может перефразировать фразу своими словами — переписать |
| Имена в коде | Свои функции урока (`predict_price`, `mae`) — допустимы; чужие API без объяснения — нет |

**Нарушение:** «Линейная модель (коэффициенты заданы, как до `fit` в sklearn)» на паре 2 модуля 1 — учащийся не знает ни `fit`, ни sklearn.

**Замена:** «Формула цены (числа задаём сами, не подбираем по данным)».

См. также Lesson Design: зона D — профконтекст для автора и учителя; зона B — подсказки **устно классу**, без нераскрытого жаргона.

---

## 5. Эксперименты

- Учащийся **меняет параметры**, сравнивает результаты, записывает наблюдения.
- Хотя бы один эксперимент — **без заранее заданного ответа** (MDS, раздел 8).
- Ритм: идея → код → наблюдение → объяснение → изменение.

---

## 6. Визуализации

- Каждый график отвечает на **конкретный вопрос**.
- Декоративные графики **не допускаются**.
- Подписи осей и заголовки — осмысленные.

---

## 7. Самостоятельная работа на паре

Каждый ноутбук **пары** заканчивается заданием, где учащийся **расширяет** решение на уроке, а не повторяет его.

**Домашнее задание** — **отдельный файл** в папке урока (`homework.ipynb`, `homework.md`, `homework.py`). Не размещать в `lesson.ipynb`. Правила — Lesson Design § «Домашнее задание»; ориентир **1 час**; уровни A–D (закрепление + вызов).

### Решения (для преподавателя)

| Правило | Пояснение |
|---|---|
| Файл | `solutions.ipynb` в папке урока |
| Содержание | Решения **всех** задач `lesson.ipynb` и `homework.ipynb` (включая вызов в ДЗ) |
| Полнота | Не «фрагмент урока»; пробел по задаче урока/ДЗ = материал не готов |
| Шапка | Первая markdown: **Для преподавателя** + что покрыто |
| Структура | Секции `## Урок. …` / `## ДЗ. …` по номерам задач |
| Комментарии | Кратко: типичные ошибки, контракты, выводы экспериментов |
| Аудитория | Только преподаватель; **не** раздавать ученикам |
| Формат | Те же правила кода, что для учебного ноутбука (§3); без вводных блоков урока |
| Canvas | ExternalUrl → Colab, элемент **не опубликован** — [08_CANVAS §11.2](08_CANVAS.md) |

## 8. Чек-лист ноутбука

- [ ] выполняется от начала до конца;
- [ ] воспроизводим (фиксированные seed, версии);
- [ ] первая ячейка — заголовок в одну строку, сразу за ней данные или импорты;
- [ ] малый датасет inline (повтор в модуле — копия в ноутбуке, не импорт); внешний файл только при большом объёме;
- [ ] нет dict и иных конструкций вне пререквизитов модуля (см. §3);
- [ ] нет вводных блоков (цель, идея, результаты, пререквизиты) и раздела «Итог»;
- [ ] читаем (короткие ячейки, понятный текст);
- [ ] текст без терминов из модулей, которые учащийся ещё не проходил;
- [ ] баланс текста и кода;
- [ ] есть эксперимент с открытым вопросом;
- [ ] есть самостоятельное задание **на паре** (расширение в конце `lesson.ipynb`);
- [ ] рабочие ячейки урока/ДЗ — stubs + assert, не предзаполненные ответы (§3 «Деятельность»);
- [ ] домашнее задание — в отдельном `homework.*`, не в этом файле; есть блок-вызов;
- [ ] `solutions.ipynb` закрывает **все** задачи урока и ДЗ (секции Урок/ДЗ, шапка для преподавателя);
- [ ] производит осмысленный артефакт.

Общий чек-лист материала — Material Design Standard, раздел 11.
