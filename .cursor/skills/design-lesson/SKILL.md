# Проектирование урока

## Роль

Один урок — одна **центральная идея** (MDS §4). Урок закрывает пробел в способностях, не «проходит тему». Один `LESSON.md` = одна пара КТП (2 ч учёта = **80 мин** проведения; ориентиры времени суммируются в 80).

## Обязательные документы

| Документ | Зачем |
|---|---|
| `docs/02_MATERIAL_DESIGN_STANDARD.md` | правила, 10 вопросов, решения учащегося |
| `docs/04_LESSON_DESIGN.md` | шаблон LESSON.md (зоны A–E, Draft 5) |
| `docs/01_PEDAGOGY.md` §7, §11 | роль преподавателя, выбор формата |
| `modules/.../UNIT.md` | место урока в модуле |

Форматы: `docs/05_NOTEBOOK_STANDARD.md`, `docs/06_PRESENTATION_STANDARD.md`.

Эталон: `modules/08_01_functions_recursion/lessons/02_function_as_mapping/` (predict), `03_parameters_and_return/` (transform введение), `04_practice_transform/` (transform отработка).

## Куда класть результат

```
modules/<module>/lessons/<NN>_<slug>/
  LESSON.md       # сценарий пары (зоны A–E)
  lesson.ipynb    # если нужен ноутбук
  homework.ipynb  # или homework.md / homework.py — обязательно
  solutions.ipynb # эталонные решения урока и ДЗ — только преподаватель
  slides.md       # только если нужно общее внимание (Presentation Standard)
```

`LESSON.md` — сценарий для преподавателя. Критерий: зоны **A–C** позволяют провести пару без устных пояснений автора; колонка «Материал» в B совпадает с ноутбуком.

## Workflow

Полный цикл (фазы 0–12): [docs/04_LESSON_DESIGN.md — полный цикл](../../docs/04_LESSON_DESIGN.md#полный-цикл-от-слота-ктп-до-canvas).

```
- [ ] 0. Контекст: КТП, UNIT §11, карта инструментов
- [ ] 1. Пройти 10 вопросов MDS §2 для этого урока
- [ ] 2. Роль: ориентация / введение / отработка / интеграция (одна пара)
- [ ] 3. Заполнить зону A (открыть, фраза, минимум сдачи, время, Canvas)
- [ ] 4. Заполнить зону B (ход): материал = раздел ноутбука / файл; на отработке ≥3 задачи одной цели
- [ ] 5. Зона C: ошибки + дифференциация
- [ ] 6. Зона D: зачем, идея, результаты, ML-контекст, решения (≥1–2), материалы, ДЗ — без повтора B
- [ ] 7. Создать только материалы из D; согласовать заголовки разделов с колонкой B
- [ ] 8. Калибровка пары (Lesson Design)
- [ ] 9. test-teacher → правки
- [ ] 10. review-edu-material (+ expert-edu-editor при необходимости)
- [ ] 11. Зона E (§13) — все поля школьной строки; согласовать с A–C
- [ ] 12. git push → gist → Canvas (Colab в отдельной вкладке; см. 08_CANVAS §11)
- [ ] 13. Фиксация: поле Canvas в A и §13
- [ ] 14. Если изменились навыки пары — строка в [матрице навыков](../../docs/ktp/08_skills_matrix.md) ([07_KTP](../../docs/07_KTP.md))
```

Шаги 1–7 — проектирование; 8–11 — качество; 12–13 — публикация.

## Ноутбук (если нужен)

Структура по Notebook Standard §2. Заголовки разделов (`## …`) = якоря в колонке «Материал» зоны B.

Пересборка ipynb: `modules/<module>/generate_notebooks.py` — править скрипт, не json вручную.

## Презентация (если нужна)

Только когда нужно **общее визуальное внимание** (Presentation Standard). Не дублировать ноутбук. **Не создавать «по умолчанию»** — если достаточно ноутбука/доски, в зоне D: «презентация не нужна».

## Data/ML

- Профессиональный контекст в зоне D — конкретный (API, метрика, pipeline).
- Примеры на данных модуля, если есть `data/module_datasets.py`.

## Правила

- Не дублировать MDS в LESSON.md — ссылки на разделы.
- ДЗ: **обязательно** на каждой паре; отдельный `homework.*`; **1 час**; уровни A–D (закрепление + вызов; MDS §6).
- Не выходить за «Вне scope» модуля (UNIT.md §14).
- Не плодить второй файл «сценарий / дизайн».

## После завершения

Сообщить: центральная идея, материалы, минимум сдачи, связь с модулем.

При сомнении в ясности — skill `test-teacher` (зоны A–C + ноутбук).
