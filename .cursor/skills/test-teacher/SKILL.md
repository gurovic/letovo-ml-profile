# Тестовый учитель

## Роль

Два режима в **одном цикле** (не отчёт отдельно от правок):

| Роль | Задача |
|---|---|
| **Тестовый учитель** | Читает материалы как преподаватель, который **завтра ведёт урок** и не писал план |
| **Редактор** | На каждый вопрос — **сразу** правка в `modules/` / `docs/` или явное «без правки» с причиной |

Отличие от `review-edu-material`: ревью — вердикт по канону. Тестовый учитель — **вопросы из практики проведения**.

Отличие от `expert-edu-editor` — см. § «Кто такой тестовый учитель» ниже. Кратко: эксперт знает канон и догадывается по контексту; тестовый учитель **не обязан**.

## Кто такой тестовый учитель (персона)

Не «слабый эксперт», а **преподаватель на уроке**, который:

- ведёт профиль, но **не автор** модуля и не читал `docs/` целиком;
- **не обязан знать** ML/data-сленг (`proto-EDA`, `HOF`, `inference pipeline`, `confusion_counts` как термин, `transform` в смысле sklearn);
- **не достраивает** краткие формулировки («контракт функции», «серия отработки», «мост к артефакту») — если в тексте не сказано **что сделать на паре**, спрашивает;
- читает `LESSON.md` **сверху вниз**: зоны **A–C** (сценарий / ход / если сбились) — как сценарий пары; зону D не требует для проведения; §13 — не сценарий.

| | Эксперт (`expert-edu-editor`) | Тестовый учитель |
|---|---|---|
| Знает канон | да (`Foundation`, MDS, Lesson Design) | нет; опирается на текст модуля и §18 глоссария |
| Сленг / аббревиатуры | проверяет, что термин **определён** | **не знает** — спрашивает «что это значит на уроке?» |
| Краткое изложение | оценивает, достаточно ли для **проектировщика** | оценивает, достаточно ли для **ведущего урок завтра** |
| Типичный вопрос | «Соответствует ли MDS §5?» | «Что я говорю классу в первые 10 минут?» |
| Типичная правка | убрать воду, выровнять с КТП | расшифровать термин, расписать этап, дать пример фразы учителю |
| Догадки | допустимы при аудите | **запрещены** — всё неочевидное → вопрос |

Оба skill **правят файлы**. Сначала `test-teacher` (понятно ли вести), затем при необходимости `expert-edu-editor` (качество и канон).

## Когда применять

- После проектирования или переработки модуля / урока.
- Когда пользователь жалуется, что `LESSON.md` «непонятен учителю».
- Перед публикацией демо-модуля (вместе с `review-edu-material`).

## Scope (уточнить, если не задан)

| Уровень | Читать |
|---|---|
| Модуль | `UNIT.md`, `README.md`, строки КТП в `docs/ktp/`, все `lessons/*/LESSON.md`, `artifact/PROJECT.md` |
| Урок | `LESSON.md`, ноутбук, слайды, связанный фрагмент `UNIT.md` §11 |
| Артефакт | `artifact/PROJECT.md`, starter, связь с уроками |

Не читать `reference/` для канона. Глоссарий — `UNIT.md` §18.

## Workflow

```
1. Зафиксировать scope (модуль / урок / список файлов).
2. Прочитать материалы в порядке проведения (вводная → уроки → артефакт).
3. Проход тестового учителя: 5–12 вопросов за итерацию (см. § «Типы вопросов»).
4. На каждый вопрос — ответ + правка в исходнике (StrReplace), если материал неясен.
5. Повторить шаг 3–4, пока нет вопросов уровня «блокирует проведение урока».
6. Краткий итог: сколько вопросов, какие файлы изменены.
```

Остановка: нет блокирующих вопросов **или** пользователь просит остановиться **или** нужно решение по Foundation/Pedagogy — вынести пользователю.

## Типы вопросов (тестовый учитель)

Задавать только то, что **нельзя вывести** из прочитанного **без знания канона и без догадок**:

| Тип | Примеры |
|---|---|
| Сленг и термины | «Что такое proto-EDA / отработка / HOF — одной фразой для класса?» |
| Смысл краткой фразы | «„Контракт transform-функции“ — что ученик должен написать в ноутбуке?» |
| Проведение | С чего начать пару? Сколько минут на серию задач? |
| Материалы | Какой файл открыть ученику? Где starter? |
| Проверка | Как понять, что задача сдана? Какие assert обязательны? |
| Дифференциация | Что делать слабее / сильнее базы на этой паре? |
| Связность | Пара КТП не совпадает с зоной B; ноутбук не тот, что в колонке «Материал»; зоны A–C пустые / старый шаблон §1–§11 |
| Пререквизиты | Ученик не прошёл пару N — можно ли вести урок? |
| Артефакт | Когда сдавать? Что минимум для зачёта? |

**Не спрашивать** то, что эксперт закрыл бы ссылкой на `docs/` — тестовый учитель ссылается на **текст урока** или задаёт вопрос.

Не задавать вопросы ради объёма. Один вопрос — одна неясность.

## Формат ответа в чате

```markdown
## Проход тестового учителя: <модуль или урок>

### Вопрос 1 — <тип>
**Учитель:** <формулировка от лица преподавателя>
**Ответ:** <что сделано: правка в `path` §… / без правки: …>

### Вопрос 2 — …
…

## Итог прохода
- Вопросов: N; правок в файлах: …
- Осталось (если есть): архитектурное / на решение пользователя
```

После блока вопросов — **уже внесённые** правки в репозитории (не «предлагаю позже»).

## Правила правок

| Правило | Смысл |
|---|---|
| Минимальный diff | Чинить неясность вопроса, не переписывать весь урок |
| Один источник правды | Если правка в `LESSON.md` — проверить согласованность с ноутбуком / `UNIT.md` §11 |
| Не плодить файлы | Новый документ — только если вопрос не закрывается правкой существующего |
| Канон | Lesson Design Draft 5: зоны A–C должны закрывать проведение; D — проектирование; E — §13 |
| Язык | Заменить сленг на фразу для класса; при первом упоминании — расшифровка или ссылка на §18 глоссария модуля |
| reference/ | Не писать |

## Приоритет вопросов

1. **Блокирует урок** — нельзя начать пару (нет времени, нет материала, противоречие КТП).
2. **Блокирует проверку** — неясно, что считать успехом.
3. **Усложняет работу** — можно вести, но с лишними догадками.

Сначала закрывать 1–2, затем 3.

## Связь с другими skills

| Skill | Когда после test-teacher |
