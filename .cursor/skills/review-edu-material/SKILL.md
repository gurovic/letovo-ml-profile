# Ревью образовательных материалов

## Роль

Главный редактор: целостность системы документов, не правка стиля ради стиля.

## Иерархия при противоречии

```
Foundation → Pedagogy → MDS → Unit Planner → Lesson Design → форматные стандарты
```

## Workflow

```
- [ ] 1. Определить тип: модуль / урок / ноутбук / презентация / артефакт
- [ ] 2. Прочитать соответствующий шаблон в docs/
- [ ] 3. Пройти чек-лист формата + MDS §11
- [ ] 4. Проверить терминологию (02_Терминология.mdc)
- [ ] 5. Проверить антипаттерны Foundation §10
- [ ] 6. Выдать вердикт и список правок по приоритету
```

## Чек-лист по типу

### Модуль (UNIT.md)

| Критерий | Вопрос |
|---|---|
| Цель | Почему программа слабее без модуля? |
| Пререквизиты | Достаточны и не избыточны? |
| Результаты | Наблюдаемые действия, не «понимает»? |
| Уроки | Каждый вносит вклад; нет «по традиции»? |
| Артефакт | Демонстрирует заявленные результаты? |
| Scope | Вне scope явно указан? |
| Траектория | Согласован с Foundation §7? |

### Урок (LESSON.md)

| Критерий | Вопрос |
|---|---|
| Сценарий A–C | Можно провести пару без догадок? Материал в B = разделы ноутбука? |
| Одна идея | Зона D — одна центральная? |
| Решения | Зона D — есть выбор ученика (не «написать код»)? |
| Активность | Учащийся работает в B, не только слушает? |
| Ошибки | Зона C — конкретные, с фразой учителя? |
| Материалы | Зона D — каждый обоснован? |
| §13 | Зона E заполнена и согласована с A–C? |
| Формат | Презентация не нужна «по умолчанию»; не дублирует ноутбук |

### Ноутбук

- выполняется сверху вниз;
- **деятельность:** stubs + assert в уроке/ДЗ, не предзаполненный прогон готовых ответов ([Notebook Standard §3](../../docs/05_NOTEBOOK_STANDARD.md));
- `solutions.ipynb` покрывает **все** задачи урока и ДЗ (не «см. урок»);
- ДЗ: закрепление + вызов;
- есть эксперимент с открытым вопросом;
- есть самостоятельное задание на паре (расширение);
- графики только с вопросом (если есть);
- нет лекционных простыней;
- глубина не ниже эталона `08_01` (пара 2) — иначе вердикт **доработать** / **перепроектировать**.

### Презентация (slides.md)

- один слайд — одна цель;
- есть роль в уроке (не «для красоты»);
- код читаем с задних рядов.

## Правила ревью

- Не предлагать новые сущности без необходимости (rules 99).
- Дублирование с MDS — **удалить**, не копировать в материал.
- Если проблема в архитектуре docs — указать документ, не чинить только симптом в модуле.
- Признаки халтуры (пустой каркас, ML-наклейка, нерабочий код, **scaffold-ноутбуки**) — вердикт **перепроектировать**; полировка → skill `expert-edu-editor`.
- Неясность для преподавателя при проведении урока — прогон → skill `test-teacher` (вопросы + правки в том же цикле).
- Статус модуля в README/`UNIT.md` — честный: **материалы** vs **план**; не смешивать.
- Не пропускать в «готов» материалы, у которых LESSON полный, а ноутбуки — тонкий Generate & Run All.

## Эталон для сравнения

`modules/08_01_functions_recursion/` — **обязательный стандарт глубины** ноутбуков и связки LESSON↔материалы для следующих модулей (не «ориентир по желанию»). Сюжет/датасет — свои; паттерн stubs / assert / solutions / вызов в ДЗ — тот же.
